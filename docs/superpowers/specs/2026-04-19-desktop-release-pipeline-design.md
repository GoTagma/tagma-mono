# Desktop Release Pipeline — Design

**Date:** 2026-04-19
**Scope:** tagma-mono + tagma-web
**Initial version:** `0.1.0`

## 1. Goal

Publish the Tagma desktop app (Electron) as GitHub Releases for Windows, Linux,
and macOS. After each release, tagma-web is automatically updated so its
homepage shows the new version and the changelog page gets a new entry with
download links. No release binaries are ever stored on the web server — all
downloads point back to GitHub Releases.

## 2. Sources of truth

### tagma-mono (this repo)

- `packages/electron/package.json` — desktop version (the authoritative version
  for GitHub Releases).
- `packages/electron/CHANGELOG/<version>.md` — hand-authored changelog entry.
  Frontmatter matches tagma-web's Astro schema (`version`, `date`, `channel`,
  `sha?`, `summary`). Body is normal markdown.
- `packages/electron/build/icon.{icns,ico,png}` — platform icons (already exist).

### tagma-web

Nothing is hand-edited for a release. The following are fully owned by CI:

- `src/content/changelog/<version>.md` — copied from tagma-mono with an
  appended "## Downloads" section.
- `src/site.config.ts` — fields `version`, `channel`, `build`, `buildDate`,
  `sha256Short`, `sizeMB` are patched in place. Other fields untouched.

## 3. Release triggers

Two ways to fire a release:

- **Tag push** (authoritative). Push a tag matching `desktop-v*` (e.g.
  `desktop-v0.1.0`). This triggers the build + publish + sync pipeline.
- **workflow_dispatch** (convenience). Run "Release Desktop" from the Actions
  UI with inputs `bump=patch|minor|major` and `channel=alpha|beta|rc|stable`.
  The workflow bumps `packages/electron/package.json`, scaffolds a
  `CHANGELOG/<new-version>.md` (with TODO placeholders), commits, tags, and
  pushes — which then re-enters the pipeline via the tag-push handler.

The `desktop-v*` prefix namespaces these tags away from plain `v*` tags that
`bun pm version` creates for npm packages (sdk/types/drivers).

## 4. Build matrix & artifact naming

A single workflow `release-desktop.yml` runs a 3-way matrix:

| Runner          | electron-builder target | Artifact filename              |
|-----------------|-------------------------|--------------------------------|
| `macos-latest`  | `dmg` (`x64` + `arm64`) | `Tagma-mac-x64.dmg`, `Tagma-mac-arm64.dmg` |
| `ubuntu-latest` | `AppImage`              | `Tagma-linux-x86_64.AppImage`  |
| `windows-latest`| `nsis`                  | `Tagma-win-x64.exe`            |

Artifact naming is fixed via `artifactName: "Tagma-${os}-${arch}.${ext}"` in
the electron-builder config. Stable filenames are what make
`releases/latest/download/<file>` URLs work.

Each installer gets a sibling `<file>.sha256` produced by a CI step before
upload.

## 5. Stable download URLs

- **Homepage buttons** (always latest):
  `https://github.com/GoTagma/tagma-mono/releases/latest/download/Tagma-mac-arm64.dmg`
  (and analogs for win/linux/mac-x64).
- **Changelog entries** (pinned to that release):
  `https://github.com/GoTagma/tagma-mono/releases/download/desktop-v0.1.0/Tagma-mac-arm64.dmg`

## 6. Code signing

Signing is wired but optional. The CI reads secrets via env vars; if a secret
is absent, electron-builder produces unsigned artifacts (no failure).

| Secret                              | Covers                          |
|-------------------------------------|---------------------------------|
| `CSC_LINK`, `CSC_KEY_PASSWORD`      | macOS Developer ID signing      |
| `APPLE_ID`, `APPLE_APP_SPECIFIC_PASSWORD`, `APPLE_TEAM_ID` | macOS notarization |
| `WIN_CSC_LINK`, `WIN_CSC_KEY_PASSWORD` | Windows code signing          |

Linux does not require signing.

## 7. Cross-repo sync

After `gh release create` succeeds, a `sync-web` job runs on `ubuntu-latest`:

1. Mint a GitHub App installation token (`RELEASE_BOT_APP_ID` +
   `RELEASE_BOT_PRIVATE_KEY`, scoped to `tagma-web` only).
2. Checkout both repos into sibling dirs.
3. Download all release artifacts (for reading sizes/checksums).
4. Run `scripts/release/sync-tagma-web.mjs`:
   - Copy `packages/electron/CHANGELOG/<version>.md` → `tagma-web/src/content/changelog/<version>.md`
   - Append a "## Downloads" section to the copy with 4 pinned download links
     + sha256 for each platform.
   - Patch `tagma-web/src/site.config.ts` using per-field regex replacements
     (no full-file rewrite): `version`, `channel`, `build` (`YYYY.MM.DD`),
     `buildDate` (`YYYY-MM-DD`), `sha256Short` (mac-arm64 hash shortened to
     `XXXX…XXXX`), `sizeMB` (mac-arm64 dmg size in MB, rounded).
5. Commit with message `chore: sync desktop v<version>` and push to `main`.

tagma-web's Cloudflare Workers deploys the rebuilt site automatically.

## 8. Release body (GitHub Releases page)

Generated by `scripts/release/build-release-notes.mjs`:

1. The hand-authored body from `CHANGELOG/<version>.md` (frontmatter stripped).
2. Appended "## Downloads" section with 4 links + sha256 + size per platform.

If the changelog's `channel` is not `stable`/`patch`, the release is marked as
pre-release.

## 9. Authentication

All writes use a GitHub App token:

- tagma-mono: creating the Release, the workflow_dispatch bump commit/tag.
- tagma-web: commit + push to `main`.

The default `GITHUB_TOKEN` in tagma-mono can remain read-only (org policy).

Required secrets in tagma-mono:
- `RELEASE_BOT_APP_ID`
- `RELEASE_BOT_PRIVATE_KEY`

The GitHub App "Tagma Release Bot" is installed on both `tagma-mono` and
`tagma-web` with `Contents: Read and write`.

## 10. Files to add / change

### New files in tagma-mono
- `.github/workflows/release-desktop.yml`
- `scripts/release/build-release-notes.mjs`
- `scripts/release/channel-info.mjs`
- `scripts/release/compute-checksums.mjs`
- `scripts/release/sync-tagma-web.mjs`
- `packages/electron/CHANGELOG/0.1.0.md`

### Changed files in tagma-mono
- `packages/electron/package.json` — set `build.artifactName`, add mac
  `arch: [x64, arm64]`, keep `signAndEditExecutable: false` default.

### tagma-web
No repo changes required up front. CI will overwrite
`src/content/changelog/*.md` and patch `src/site.config.ts` on each release.
Existing placeholder 0.8.x changelog entries can be deleted once the 0.1.0
release lands — out of scope for this design.

## 11. Out of scope

- **Auto-updater** (electron-updater): `latest.yml` / `latest-mac.yml` are
  uploaded so we can enable it later without breaking existing installs; no
  in-app update UI in this iteration.
- **`tagma-cli` releases**: a separate repo, separate pipeline.
- **npm publish** for sdk/types/drivers: already handled by existing
  `publish:*` scripts, unaffected by this work.

## 12. Rollback

- A bad desktop release can be deleted from the GitHub Releases UI; the
  `/latest/download/...` URLs immediately fall back to the previous release.
- tagma-web's changelog entry and `site.config.ts` bump are a single commit on
  `main`; revert the commit to restore the previous shipping version on the
  homepage.
